# Plan de Optimización Streamlit/Athena para Fargate

## Objetivo
Reducir latencia, coste y consumo de memoria en la app Streamlit que consulta datos Gold en Athena, restringiendo la interacción a particiones existentes y evitando consultas innecesarias. La solución debe funcionar en una tarea Fargate pequeña sin colgarse.

## Contexto Observado
- `app.py` usa `st.date_input` para búsqueda por ID y exploración, permitiendo cualquier fecha aunque solo existan particiones para los días 4, 5 y 6.
- El explorador ejecuta hasta cuatro consultas Athena por partición: entidades, medios, decisiones y datos filtrados.
- `explorar_solicitudes()` se ejecuta automáticamente en cada rerun de Streamlit, incluso mientras el usuario cambia filtros.
- `athena_client.py` valida formato de fecha, pero no valida que la fecha exista como partición.
- `ORDER BY fecha_solicitud DESC LIMIT N` puede ser caro en Athena si la partición crece.
- No hay `.dockerignore`; con `.env` y `.venv` en el workspace, el contexto de build puede ser pesado o sensible.

## Estrategia
Priorizar cambios de alto impacto y bajo riesgo:
1. Reducir consultas Athena por interacción.
2. Restringir fechas desde UI y capa de datos.
3. Cachear resultados estables por partición.
4. Evitar trabajo automático en reruns.
5. Preparar despliegue Fargate con menor memoria base y menor riesgo operativo.

## Plan de Implementación

### 1. Centralizar particiones disponibles
Crear una fuente única para las particiones válidas.

Opción recomendada inicial:
- Definir `PARTICIONES_VALIDAS` configurable por variable de entorno, por ejemplo `AVAILABLE_PARTITIONS=2026-05-04,2026-05-05,2026-05-06`.
- Exponer helpers en `athena_client.py`:
  - `obtener_particiones_validas() -> list[str]`
  - `validar_particion(fecha_dia: str) -> str`

Tradeoff:
- Lista por entorno: más rápida, cero query adicional, ideal para demo/Fargate pequeño.
- Descubrir particiones desde Athena: más flexible, pero agrega consulta y complejidad. Dejarlo como mejora futura si las particiones cambian dinámicamente.

### 2. Reemplazar calendarios abiertos por selectores
Modificar `app.py` para reemplazar ambos `st.date_input` por `st.selectbox` usando las particiones válidas.

Cambios esperados:
- Búsqueda por ID: selector `Día de partición` limitado a 4, 5 y 6.
- Explorador: selector `Día a explorar` limitado a las mismas fechas.
- Convertir la fecha seleccionada directamente a string, sin permitir fechas inexistentes.

Resultado:
- Se elimina la posibilidad de consultar particiones vacías por error del usuario.
- Se simplifica la UI y se reduce ruido operativo.

### 3. Validar partición también en la capa Athena
Actualizar `validar_fecha_dia()` o crear `validar_particion()` para rechazar fechas que no estén en `PARTICIONES_VALIDAS`.

Motivo:
- La UI guía, pero la capa de datos debe proteger la regla de negocio.
- Evita llamadas internas o futuras rutas que consulten fechas inválidas.

### 4. Evitar consultas automáticas del explorador en cada rerun
Encapsular filtros del explorador en `st.form()`.

Cambios esperados:
- Renderizar filtros sin ejecutar `explorar_solicitudes()` inmediatamente.
- Ejecutar Athena solo al pulsar un botón tipo `Consultar solicitudes`.
- Guardar el resultado en `st.session_state` para mantener la tabla visible entre reruns.

Resultado:
- Streamlit deja de consultar Athena mientras el usuario ajusta filtros.
- Se reduce latencia percibida y coste por interacción.

### 5. Unificar consultas de valores de filtros
Reemplazar las tres funciones cacheadas actuales (`cargar_entidades`, `cargar_medios`, `cargar_decisiones`) por una sola función cacheada por partición.

Nueva función sugerida en `athena_client.py`:
- `obtener_opciones_filtros(fecha_dia: str) -> dict[str, list[str]]`

SQL sugerido:
```sql
SELECT
  array_sort(array_agg(DISTINCT entidad_emisora)) AS entidades,
  array_sort(array_agg(DISTINCT medio_emisor)) AS medios,
  array_sort(array_agg(DISTINCT decision_cobertura)) AS decisiones
FROM database.table
WHERE fecha_dia = DATE 'YYYY-MM-DD'
```

Consideración:
- Si Athena/Presto devuelve tipos complejos incómodos vía `awswrangler`, usar alternativa con tres subqueries agregadas o mantener tres queries como fallback. La meta es una query por partición, no tres.

### 6. Cachear consultas estables
Agregar wrappers con `@st.cache_data` en `app.py`:
- Búsqueda por ID: cache por `(id_solicitud, fecha_dia)`.
- Opciones de filtros: cache por `fecha_dia`.
- Exploración: cache por `(fecha_dia, entidad, medio, decision, limite)` con TTL moderado.

TTL recomendado:
- Para particiones históricas inmutables: `ttl=3600` o más.
- Si los datos todavía se reprocesan: `ttl=300`.

### 7. Revisar `ORDER BY` en exploración
Mantener inicialmente `ORDER BY fecha_solicitud DESC LIMIT N` si la UX necesita “últimas solicitudes”.

Optimización opcional:
- Agregar una constante/env `EXPLORER_ORDER_BY_ENABLED=true`.
- Permitir desactivarlo para modo bajo coste.

Tradeoff:
- Con `ORDER BY`: resultados más útiles, pero Athena puede ordenar una partición grande.
- Sin `ORDER BY`: menor coste/latencia, pero resultados menos previsibles.

### 8. Mejorar seguridad y peso del build Docker
Crear `.dockerignore` con al menos:
```dockerignore
.env
.venv
.git
__pycache__
*.pyc
.pytest_cache
.cache
```

Motivo:
- Docker no respeta `.gitignore`.
- Evita enviar `.env`, `.venv` y `.git` al contexto de build.
- Reduce tiempo de build y riesgo de secretos.

### 9. Ajustes recomendados para Fargate
Sin cambiar arquitectura todavía:
- Definir `STREAMLIT_BROWSER_GATHER_USAGE_STATS=false`.
- Usar health check contra `/_stcore/health`.
- Considerar tarea mínima de `0.5 vCPU / 1 GiB` si se mantiene `awswrangler`, `pandas` y `pyarrow`.
- Si se exige `0.25 vCPU / 512 MiB`, medir memoria real y evaluar reemplazar `awswrangler.read_sql_query` por cliente Athena con `boto3` paginado y conversión más controlada.

## Orden de Trabajo Recomendado
1. Implementar particiones válidas configurables y validación doble.
2. Cambiar `date_input` por `selectbox`.
3. Mover explorador a `st.form()` y usar `st.session_state`.
4. Unificar/cachar opciones de filtros.
5. Cachear búsqueda y exploración.
6. Agregar `.dockerignore`.
7. Validar manualmente flujos principales sin ejecutar build.

## Validación
- Verificar que la UI solo muestra días 4, 5 y 6 según `AVAILABLE_PARTITIONS`.
- Verificar que una fecha fuera de la lista falla desde `athena_client.py` aunque se llame directamente.
- Confirmar que cambiar filtros no dispara Athena hasta pulsar `Consultar solicitudes`.
- Confirmar que las opciones de filtro se consultan una sola vez por partición dentro del TTL.
- Confirmar que búsqueda repetida por ID usa cache durante el TTL.
- Confirmar que `.dockerignore` excluye `.env`, `.venv` y `.git`.

## Riesgos
- Si las fechas reales tienen otro año/mes, `AVAILABLE_PARTITIONS` debe configurarse correctamente en el entorno de despliegue.
- La agregación con `array_agg(DISTINCT ...)` puede necesitar ajuste según tipos devueltos por Athena/awswrangler.
- Cachear demasiado tiempo puede mostrar datos obsoletos si las particiones 4, 5 y 6 se reprocesan.
- El tamaño mínimo viable de Fargate depende de memoria real en runtime; `awswrangler + pandas + pyarrow` difícilmente será cómodo en 512 MiB.

## Fuera de Alcance Inicial
- Reescribir el cliente Athena con `boto3` puro.
- Crear vistas/materialized views o tablas auxiliares en AWS.
- Cambiar el pipeline de particionado.
- Rediseñar visualmente toda la aplicación.
