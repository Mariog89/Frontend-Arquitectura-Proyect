# Plan: Refactor frontend Gold S3 + Athena

## Objetivo

Refactorizar el frontend Streamlit para leer indicadores agregados directamente desde Parquet en S3 y optimizar la búsqueda puntual de solicitudes mediante la tabla Gold `solicitudes_lookup_por_id`.

El frontend debe mantener:

- Búsqueda por `id_solicitud` con cache y lectura puntual optimizada.
- Exploración paginada de solicitudes cuando siga siendo necesaria.
- Panel de indicadores con tabs para:

- Cobertura por perfil.
- Costos por entidad.
- Total de órdenes por día.
- Trazabilidad por entidad como tabla.

## Diagnóstico Actual

El proyecto es una app Streamlit pequeña con dos archivos principales:

- `app.py`: UI, cache Streamlit, búsqueda por ID y explorador paginado.
- `athena_client.py`: consultas Athena vía `awswrangler`, validación de particiones y construcción SQL.

La app actual todavía asume el esquema Gold anterior:

- Extrae paciente, edad, perfil y medicación desde `paciente_info` con `regexp_extract`.
- Deriva `decision_cobertura` desde `CAST(cobertura AS BOOLEAN)`.
- No conoce `entidad_emisora`.

Las consultas nuevas producen un Gold más limpio:

- `ordenes_clasificadas` ya trae columnas normalizadas: `paciente_nombre`, `paciente_dno`, `paciente_seguro_social`, `edad`, `perfil_cobertura`, `medicacion`, `decision_cobertura`, `entidad_emisora`, etc.
- Existen tablas agregadas ya materializadas en Gold: `trazabilidad_por_entidad`, `total_ordenes_por_dia`, `cobertura_por_perfil`, `costos_por_entidad`.
- Existe una nueva tabla lookup optimizada para búsqueda puntual: `solicitudes_lookup_por_id`, particionada por `id_bucket`.

## Decisión Actualizada

El frontend NO debe leer los indicadores agregados desde Athena. Debe leerlos directamente desde S3 usando `wr.s3.read_parquet()` con cache Streamlit.

La búsqueda por ID debe usar la tabla lookup `s3://coil-arquitectura/gold/solicitudes_lookup_por_id/`, calculando el bucket localmente con `zlib.crc32(id_solicitud) % 100`. Esto evita escanear todas las particiones de `ordenes_clasificadas`.

Athena queda como mecanismo complementario solo si se decide preservar el explorador paginado sobre `ordenes_clasificadas`. La búsqueda puntual por ID ya no debe recorrer `fecha_dia IN (...)`.

## Riesgos Detectados

1. El frontend se romperá contra el nuevo Gold si sigue usando `paciente_info` y `cobertura`.
2. La búsqueda por usuario ya no debe consultar todas las particiones de `ordenes_clasificadas`; debe leer el bucket correspondiente en `solicitudes_lookup_por_id`.
3. `AVAILABLE_PARTITIONS` está hardcodeado por variable de entorno; si Athena/S3 gana nuevas particiones, el frontend no las ve automáticamente.
4. `costos_por_entidad` está particionada por `fecha_dia`, pero las otras tablas agregadas no lo están según tus CTAS.
5. El listing de S3 muestra dos archivos por partición en `gold/ordenes_clasificadas`. Si eso viene de recrear CTAS sin limpiar el prefijo S3, Athena puede leer duplicados. Antes de confiar en métricas, hay que verificar si la tabla apunta a todos esos objetos o solo a una definición consistente.
6. Los nombres reales de la tabla `cobertura_por_perfil` en tu SQL son `porcentaje_cobertura_aplicado` y `monto_total_cubierto`, pero el CSV local tiene `porcentaje_cobertura_catalogo` y menos columnas. El frontend debe seguir el contrato actual de Athena, no el CSV viejo.
7. El hash de bucket debe coincidir EXACTAMENTE entre Athena y Python. La expresión Athena usa `MOD(crc32(to_utf8(CAST(id_solicitud AS VARCHAR))), 100)`; Python debe usar `zlib.crc32(id_solicitud.encode("utf-8")) & 0xFFFFFFFF` antes del módulo.
8. `wr.s3.read_parquet(path=..., dataset=True)` sobre un prefijo de bucket inexistente puede fallar dependiendo de cómo responda S3/AWS Wrangler. La implementación debe manejar `NoFilesFound` o errores equivalentes y devolver un `DataFrame` vacío.

### 2. Preservar búsqueda por ID con lookup S3 particionado por bucket

Nuevo comportamiento esperado:

- El usuario ingresa `id_solicitud`.
- El backend valida el ID.
- El backend calcula `id_bucket = crc32(id_solicitud) % 100`.
- El frontend lee solo `s3://coil-arquitectura/gold/solicitudes_lookup_por_id/id_bucket=<bucket>/`.
- Se filtra en memoria por `id_solicitud` dentro de ese bucket.
- Se devuelve `head(1)`.
- La UI sigue mostrando datos del paciente, solicitud, cobertura y trazabilidad.

Funciones a implementar o adaptar:

- `validar_id_solicitud(id_solicitud: str) -> str`.
- `calcular_id_bucket(id_solicitud: str) -> int`.
- `buscar_solicitud_por_id_s3(id_solicitud: str) -> pd.DataFrame`.

Wrapper cacheado en Streamlit:

```python
@st.cache_data(ttl=3600)
def buscar_solicitud_cached(id_solicitud: str):
    return buscar_solicitud_por_id_s3(id_solicitud)
```

Tradeoff:

- Ventaja: evita escanear `ordenes_clasificadas` completa y reduce costo/latencia.
- Costo: cada búsqueda lee un bucket completo. Con 100 buckets y deduplicación por ID, es aceptable para búsqueda puntual.
- Requisito: el CTAS lookup debe recrearse después de cambios en `ordenes_clasificadas` para no servir datos obsoletos.

## Contrato de Lookup por ID

La tabla `coil_gold.solicitudes_lookup_por_id` está optimizada para búsqueda puntual:

- Ubicación: `s3://coil-arquitectura/gold/solicitudes_lookup_por_id/`.
- Partición: `id_bucket`.
- Buckets: `0..99`.
- Dedupe: `ROW_NUMBER() OVER (PARTITION BY id_solicitud ORDER BY timestamp_procesamiento DESC) = 1`.
- Hash Athena: `MOD(crc32(to_utf8(CAST(id_solicitud AS VARCHAR))), 100)`.
- Hash Python: `zlib.crc32(id_solicitud.encode("utf-8")) & 0xFFFFFFFF`, luego `% 100`.

Código base esperado:

```python
import re
import zlib

import awswrangler as wr
import pandas as pd


LOOKUP_BASE_PATH = "s3://coil-arquitectura/gold/solicitudes_lookup_por_id/"
TOTAL_BUCKETS = 100


def validar_id_solicitud(id_solicitud: str) -> str:
    id_solicitud = id_solicitud.strip()

    if not re.match(r"^[A-Za-z0-9_\-]+$", id_solicitud):
        raise ValueError("ID de solicitud inválido.")

    return id_solicitud


def calcular_id_bucket(id_solicitud: str) -> int:
    checksum = zlib.crc32(id_solicitud.encode("utf-8")) & 0xFFFFFFFF
    return checksum % TOTAL_BUCKETS


def buscar_solicitud_por_id_s3(id_solicitud: str) -> pd.DataFrame:
    id_solicitud = validar_id_solicitud(id_solicitud)
    id_bucket = calcular_id_bucket(id_solicitud)

    path = f"{LOOKUP_BASE_PATH}id_bucket={id_bucket}/"
    df = wr.s3.read_parquet(path=path, dataset=True)

    if df.empty:
        return df

    resultado = df[df["id_solicitud"].astype(str) == id_solicitud].copy()
    return resultado.head(1)
```
